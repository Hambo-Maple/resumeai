# 简历助手测试输入用例

整理日期：2026-06-07  
用途：验证当前“左侧对话 + 右侧简历预览”的对话式简历教练 MVP。

## 1. 测试目标

验证完整流程是否跑通：

```text
岗位分析
-> 输入经历
-> 生成简历初稿
-> 继续优化
-> 回答追问
-> 生成新版本
-> 确认最终版
```

当前系统已切换为真实模型模式，因此测试重点是：

- 页面流程是否顺畅。
- 左侧对话是否能推进。
- 右侧简历预览是否同步变化。
- 简历状态是否从 `draft` 变为 `revised`，最后变为 `final`。

```text

1. 负责业务数据取数、清洗、分析和可视化报表搭建；

在左侧“教练追问”区域选择第一个问题。

```

### 7.3 操作

点击：

```text
发送
```

### 7.4 预期现象

左侧对话中，系统提示经历已更新。

右侧简历预览中，经历卡片应发生变化，补充内容可能进入：

- role
- actions
- metrics
- results

## 11. 验证重点清单

测试时重点看：

- 左侧是否像对话流程，而不是普通表单流程。
- 用户输入经历后，右侧是否出现经历卡片。
- 点击“生成初稿”后，右侧是否出现 `v1 · draft`。
- 点击“继续优化”后，左侧是否出现教练追问。
- 回答追问后，右侧经历卡片是否更新。
- 点击“生成新版本”后，右侧是否出现 `v2 · revised`。
- 点击“确认最终版”后，右侧是否出现 `final`。

## 12. 当前已知限制

当前使用真实大模型：

- 输出质量会受到模型稳定性、Prompt 质量和用户输入完整度影响。
- 每次生成可能存在轻微差异，需要通过评审、修复和结构化校验保证质量。
- 测试时会消耗模型调用额度。

后续接入真实模型后，需要重点验证：

- Prompt 输出是否稳定合法 JSON。
- 是否严格避免虚构经历和数据。
- 追问是否足够具体。
- 简历 bullet 是否真实、自然、有岗位针对性。

### 13.6 预期现象

新增职位后：

- 职位列表顶部出现新职位。
- 标题显示职位名称。
- 副标题显示公司、城市、状态。
- 后端返回内容包含 `id`、`createdAt`、`updatedAt`。

编辑职位后：

- 修改状态为 `applied` 或 `interviewing` 后，列表状态同步变化。
- 修改关键词或要求后，重新打开编辑态能看到最新内容。

删除职位后：

- 当前职位从列表中消失。
- 后端删除接口返回：

```json
{"deleted": true}
```

### 13.7 验证重点清单

- 职位字段中只有“职位”是必填，缺失职位时不能提交。
- `status` 只允许 `interested`、`applied`、`interviewing`、`offered`、`rejected`、`closed`。
- 关键词和要求应保存为数组，而不是一整段字符串。
- `userProfileId` 为空时也可以创建职位。
- 同一职位经过新增、编辑、删除后，列表和数据库状态一致。
