关联文档：

- docs/对话式简历教练设计.md

## 1. 当前实现状态

目前系统已经完成三条基础闭环：

```text
岗位分析
-> 经历结构化
-> 简历初稿生成
```

已实现接口：

```text
POST /api/resume/analyze-job
POST /api/resume/structure-experience
POST /api/resume/generate
```

已实现数据表：

```text
resume_tasks
job_targets
core_abilities
experiences
resume_drafts
evaluations
prompt_runs
```

已实现 Prompt：

```text
prompts/resume/analyze_job.v1.md
prompts/resume/structure_experience.v1.md
prompts/resume/generate_resume.v1.md
```

当前还缺少：

- 对话式追问。
- 用户补充信息后更新经历卡片。
- 基于补充信息生成新版本简历。
- 简历版本状态管理。
- 最终稿确认。

## 2. 实施目标

把当前的“简历生成器”升级为“对话式简历教练”。

目标体验：

```text
用户生成简历初稿
-> 系统指出当前信息缺口
-> 系统提出具体追问
-> 用户回答
-> 系统更新经历素材
-> 系统生成新版本简历
-> 用户继续优化或确认最终版
```

MVP 阶段先不做复杂聊天系统，只实现一轮清晰可用的对话优化闭环。

## 3. 推荐实施路径

建议分 5 个阶段实现。

```text
阶段一：补充简历版本状态
阶段二：实现缺口诊断和追问
阶段三：实现用户回答后的经历更新
阶段四：实现基于对话的新版本简历生成
阶段五：实现最终稿确认
```

## 4. 阶段一：补充简历版本状态

### 4.1 目标

让系统能够区分：

```text
初稿
修改稿
最终稿
```

当前 `resume_drafts` 只有 `version`，建议补充：

```text
status
parent_draft_id
change_summary
```

### 4.2 数据库调整

修改 `resume_drafts` 表：

```text
status: draft | revised | final
parent_draft_id: UUID | null
change_summary: text | null
```

字段含义：

- `status = draft`：系统第一次生成的初稿。
- `status = revised`：基于用户补充信息生成的修改稿。
- `status = final`：用户确认的最终稿。
- `parent_draft_id`：当前版本来源于哪一版。
- `change_summary`：本次修改说明。

### 4.3 后端任务

新增 Alembic 迁移：

```text
add_resume_draft_version_fields
```

修改模型：

```text
backend/app/models/resume.py
```

修改生成逻辑：

```text
ResumeService.generate_resume()
```

生成初稿时：

```text
version = 1
status = draft
parent_draft_id = null
change_summary = null
```

### 4.4 验收标准

- 新生成的简历草稿状态为 `draft`。
- 数据库中能看到 `status`、`parent_draft_id`、`change_summary` 字段。

## 5. 阶段二：实现缺口诊断和追问

### 5.1 目标

用户生成初稿后，点击：

```text
继续优化
```

系统分析当前简历和目标岗位之间的缺口，并提出下一轮追问。

### 5.2 新增 Prompt

新增：

```text
prompts/resume/diagnose_resume_gap.v1.md
```

Prompt 输入：

```text
目标公司
目标岗位
岗位核心能力
岗位关键词
结构化经历
当前简历草稿
```

Prompt 输出：

```json
{
  "diagnosis": "整体诊断",
  "gaps": [
    {
      "type": "missing_metric",
      "priority": "high",
      "description": "缺少活动转化数据",
      "relatedExperienceId": "...",
      "relatedAbility": "数据分析"
    }
  ],
  "questions": [
    {
      "question": "这次活动报名人数、到场人数分别是多少？",
      "reason": "用于补充结果量化信息",
      "relatedExperienceId": "...",
      "priority": "high"
    }
  ]
}
```

### 5.3 新增后端接口

新增：

```text
POST /api/resume/diagnose
```

请求：

```json
{
  "taskId": "...",
  "jobTargetId": "...",
  "resumeDraftId": "..."
}
```

响应：

```json
{
  "diagnosis": "...",
  "gaps": [],
  "questions": []
}
```

### 5.4 新增服务

可以新增：

```text
backend/app/services/optimization_service.py
```

或先放在：

```text
backend/app/services/resume_service.py
```

建议单独新增 `OptimizationService`，后续更清晰。

### 5.5 前端任务

在简历初稿区域增加按钮：

```text
继续优化
```

点击后展示：

- 当前诊断。
- 主要缺口。
- 系统追问问题。

### 5.6 验收标准

- 用户点击“继续优化”后，系统能输出 3-5 个具体追问。
- 追问不是泛泛的“请补充更多”，而是围绕数据、角色、行动、结果等具体信息。
- 本次诊断写入 `prompt_runs`。

## 6. 阶段三：用户回答后更新经历卡片

### 6.1 目标

用户回答系统追问后，系统不是只保存聊天文本，而是要更新结构化经历卡片。

### 6.2 新增 Prompt

新增：

```text
prompts/resume/update_experience_from_dialogue.v1.md
```

Prompt 输入：

```text
原结构化经历
系统追问
用户回答
目标岗位能力
```

Prompt 输出：

```json
{
  "experienceId": "...",
  "updatedFields": {
    "role": "...",
    "actions": [],
    "results": [],
    "metrics": [],
    "matchedAbilities": [],
    "missingInfoQuestions": []
  },
  "updateSummary": "用户补充了报名人数、到场人数和宣传渠道。"
}
```

### 6.3 新增后端接口

新增：

```text
POST /api/resume/update-experience
```

请求：

```json
{
  "taskId": "...",
  "experienceId": "...",
  "question": "这次活动报名人数、到场人数分别是多少？",
  "answer": "报名 260 人，到场 210 人。"
}
```

响应：

```json
{
  "experienceId": "...",
  "updatedExperience": {},
  "updateSummary": "..."
}
```

### 6.4 数据更新规则

更新字段：

- `role`
- `background`
- `actions`
- `results`
- `metrics`
- `matched_abilities`
- `missing_info_questions`

保留字段：

- `raw_text`
- 原始经历不删除。

### 6.5 前端任务

在追问问题下方提供回答框。

用户输入回答后点击：

```text
提交补充信息
```

系统返回更新后的经历卡片。

### 6.6 验收标准

- 用户回答追问后，经历卡片内容会更新。
- 新增数据不会覆盖原始经历。
- 本次更新写入 `prompt_runs`。

## 7. 阶段四：基于对话生成新版本简历

### 7.1 目标

用户补充信息后，系统基于更新后的经历生成新版本简历。

```text
ResumeDraft v1
-> 用户补充信息
-> ResumeDraft v2
```

### 7.2 新增 Prompt

新增：

```text
prompts/resume/rewrite_resume_from_dialogue.v1.md
```

也可以先复用已有：

```text
prompts/resume/generate_resume.v1.md
```

MVP 建议先复用 `generate_resume.v1.md`，通过接口参数标记：

```text
parentDraftId
changeSummary
```

### 7.3 修改接口

扩展：

```text
POST /api/resume/generate
```

新增请求字段：

```text
parentDraftId?: UUID
changeSummary?: string
```

如果存在 `parentDraftId`：

```text
status = revised
version = parent.version + 1
parent_draft_id = parentDraftId
change_summary = changeSummary
```

如果不存在：

```text
status = draft
version = 1
```

### 7.4 前端任务

用户提交补充信息并更新经历后，页面提供：

```text
生成新版本
```

生成后展示：

```text
简历初稿 v2
```

并显示：

```text
本次修改：补充了活动报名人数和到场人数，强化结果量化表达。
```

### 7.5 验收标准

- 能生成 `ResumeDraft v2`。
- `v2.parent_draft_id = v1.id`。
- `v2.status = revised`。
- 前端能看到新版本内容。

## 8. 阶段五：实现最终稿确认

### 8.1 目标

用户满意某一版本后，可以将其标记为最终稿。

### 8.2 新增接口

新增：

```text
POST /api/resume/finalize
```

请求：

```json
{
  "resumeDraftId": "..."
}
```

响应：

```json
{
  "resumeDraftId": "...",
  "status": "final"
}
```

### 8.3 后端规则

- 将当前 `resume_draft.status` 设置为 `final`。
- 同一任务下其他 draft 可以保持原状态，或标记为 `archived`。MVP 暂不需要 `archived`。

### 8.4 前端任务

在每个简历版本上提供：

```text
确认最终版
```

点击后显示：

```text
已确认最终版
```

### 8.5 验收标准

- 用户可以确认最终稿。
- 数据库中该版本状态变为 `final`。

## 9. MVP 推荐优先级

### P0：必须先做

1. `resume_drafts` 增加版本状态字段。
2. `diagnose_resume_gap.v1.md`。
3. `/api/resume/diagnose`。
4. 前端“继续优化”按钮和追问展示。

### P1：完成对话优化闭环

1. `update_experience_from_dialogue.v1.md`。
2. `/api/resume/update-experience`。
3. 用户回答追问后更新经历卡片。
4. 基于更新经历生成 `ResumeDraft v2`。

### P2：最终稿能力

1. `/api/resume/finalize`。
2. `status = final`。
3. 前端最终版确认。

## 10. 建议不要现在做的事

当前阶段暂不建议做：

- 完整聊天消息表。
- 多会话管理。
- Word/PDF 导出。
- 多模板排版。
- 复杂评分体系。
- 多智能体框架。

原因：

```text
先跑通一轮“追问 -> 补充 -> 更新经历 -> 生成新版本”的真实用户闭环。
```

等这个闭环稳定后，再引入完整对话消息表和版本管理后台。

## 11. 实施后的目标流程

完成本计划后，用户流程应变成：

```text
输入目标公司和岗位
-> 系统分析岗位能力
-> 用户输入经历
-> 系统结构化经历并追问
-> 用户补充信息
-> 系统生成简历初稿
-> 用户点击继续优化
-> 系统诊断缺口并追问
-> 用户回答
-> 系统更新经历卡片
-> 系统生成新版本简历
-> 用户确认最终稿
```

## 12. 第一项开发任务建议

下一步建议立即实现：

```text
diagnose_resume_gap.v1.md
POST /api/resume/diagnose
前端“继续优化”按钮
```

这是对话式简历教练的入口。它不要求马上做完整聊天系统，但能让用户第一次感受到系统在“教练式追问”，而不是只生成一段文字。
