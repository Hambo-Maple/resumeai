## 1. 架构结论

本项目采用**严格前后端分离架构**。

技术栈明确为：

- 前端：React + Vite
- 后端：FastAPI
- 数据库：PostgreSQL
- 数据库启动：Docker Compose
- ORM：SQLAlchemy
- 数据库迁移：Alembic
- Prompt 管理：Markdown 文件
- 大模型调用：后端统一封装 LLMService

不采用 Next.js API Routes 作为后端，也不采用 Next.js 全栈一体化方案。

## 2. 架构目标

简历助手智能体当前阶段的架构目标是：在保持开发速度的同时，让前端、后端、Prompt、数据库和大模型调用形成清晰边界，为后续面试模拟、能力评估、公司信息增强、文件解析、语音面试和异步任务预留空间。

首版优先保证：

- 真实用户可以完成一次岗位定制简历生成。
- 前端只调用后端业务 API，不直接调用大模型。
- 后端统一封装业务逻辑、Prompt 渲染和模型调用。
- Prompt 文件可版本化管理。
- 每次大模型调用都写入 `prompt_runs`，方便追踪、复现和调试。
- 数据模型能支撑后续扩展。

## 3. 总体架构

```text
React + Vite 前端
  |
  | HTTP 调用业务 API
  v
FastAPI 后端
  |
  | 调用业务服务
  v
Services 业务服务层
  |
  | 编排 Prompt 和模型调用
  v
PromptService + LLMService
  |
  | 读取 Prompt 模板 / 调用大模型 / 写入 PromptRun
  v
PostgreSQL + prompts 文件夹
```

### 3.1 分层说明

| 层级 | 职责 |
| --- | --- |
| React 前端 | 用户界面、表单输入、流程引导、结果展示 |
| FastAPI API 层 | 对前端暴露稳定业务接口 |
| Services 层 | 处理业务逻辑和模块编排 |
| PromptService | 加载 Prompt 模板、填充变量、管理 Prompt 版本 |
| LLMService | 统一调用大模型，记录 PromptRun |
| 数据层 | 保存任务、岗位、经历、简历草稿、评分、PromptRun |
| prompts 文件夹 | 保存给模型使用的 Prompt 模板 |

## 5. 前端架构

### 5.1 前端职责

前端只负责用户体验和状态展示，不包含核心 AI 逻辑。

主要职责：

- 创建简历任务。
- 输入目标公司、岗位和 JD。
- 展示岗位能力分析结果。
- 采集和编辑用户经历。
- 展示简历生成结果。
- 展示评分和优化建议。
- 调用后端 API。

### 5.2 前端页面

MVP 页面：

```text
/resume/new
/resume/analyze
/resume/experiences
/resume/result
```

### 5.3 前端 API 调用

前端通过统一 API client 调用后端：

```text
frontend/src/services/resumeApi.ts
```

示例：

```ts
export async function analyzeJob(payload: AnalyzeJobRequest) {
  const response = await fetch(`${import.meta.env.VITE_API_BASE_URL}/api/resume/analyze-job`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(payload),
  });

  if (!response.ok) {
    throw new Error("岗位分析失败");
  }

  return response.json();
}
```

## 6. 后端架构

### 6.1 后端职责

FastAPI 后端负责：

- 对前端暴露业务 API。
- 参数校验和响应格式控制。
- 调用业务 Service。
- 读取和渲染 Prompt。
- 调用大模型。
- 写入 PostgreSQL。
- 记录 PromptRun。

### 6.2 API 模块

首版实现：

```text
POST /api/resume/analyze-job
POST /api/resume/structure-experience
POST /api/resume/generate
POST /api/resume/evaluate
POST /api/resume/rewrite
GET  /api/resume/tasks
GET  /api/resume/drafts/{draft_id}
```

后续扩展：

```text
/api/interview
/api/assessment
/api/company
```

### 6.3 调用链

```text
React 页面
-> FastAPI Router
-> ResumeService / JobService / ExperienceService
-> PromptService.render()
-> LLMService.generate()
-> 写入 prompt_runs
-> 写入业务表
-> 返回结构化 JSON
```

## 7. Services 设计

### 7.1 ResumeService

职责：

- 创建简历任务。
- 生成简历草稿。
- 改写简历内容。
- 保存简历版本。

### 7.2 JobService

职责：

- 分析岗位 JD。
- 生成岗位能力模型。
- 提取岗位关键词。
- 生成简历重点建议。

### 7.3 ExperienceService

职责：

- 接收用户原始经历。
- 结构化经历素材。
- 生成追问问题。
- 标注经历与岗位能力的匹配关系。

### 7.4 EvaluationService

职责：

- 根据岗位能力评估简历。
- 输出评分、优势、短板和建议。

### 7.5 PromptService

职责：

- 从 `prompts/` 读取 Prompt 模板。
- 根据 `prompt_name` 和 `prompt_version` 定位模板。
- 填充变量生成 `rendered_prompt`。
- 管理 Prompt 版本。

### 7.6 LLMService

职责：

- 统一封装大模型调用。
- 处理模型参数。
- 解析模型输出。
- 记录调用耗时、token、成本。
- 写入 `prompt_runs` 表。

## 9. PromptRun 设计

### 9.1 定位

`PromptRun` 是每次大模型调用的运行记录，存入 PostgreSQL 的 `prompt_runs` 表。

它不是 Prompt 模板，也不是开发文档，而是系统运行时自动生成的调用日志。

### 9.2 记录时机

每次调用大模型都应记录一条 `PromptRun`。

例如：

```text
/api/resume/analyze-job          -> prompt_runs +1
/api/resume/structure-experience -> prompt_runs +1
/api/resume/generate             -> prompt_runs +1
/api/resume/evaluate             -> prompt_runs +1
```

### 9.3 表结构

PostgreSQL 建表示例：

```sql
CREATE TABLE prompt_runs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  user_id UUID NULL,
  task_id UUID NULL,

  module TEXT NOT NULL,
  prompt_name TEXT NOT NULL,
  prompt_version TEXT NOT NULL,
  model TEXT NOT NULL,

  input JSONB NOT NULL,
  rendered_prompt TEXT NOT NULL,
  output JSONB NULL,

  status TEXT NOT NULL DEFAULT 'success',
  error_message TEXT NULL,

  latency_ms INTEGER NULL,
  token_input INTEGER NULL,
  token_output INTEGER NULL,
  cost_estimate NUMERIC(10, 6) NULL,

  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

建议索引：

```sql
CREATE INDEX idx_prompt_runs_task_id ON prompt_runs(task_id);
CREATE INDEX idx_prompt_runs_prompt ON prompt_runs(module, prompt_name, prompt_version);
CREATE INDEX idx_prompt_runs_created_at ON prompt_runs(created_at);
```

### 9.4 隐私注意

正式环境中，`PromptRun` 可能包含用户简历、经历、手机号、邮箱等敏感信息。后续需要考虑：

- 用户数据权限隔离。
- 内部调试页面鉴权。
- 敏感字段脱敏。
- 数据保留周期。
- 用户删除数据时同步删除相关 PromptRun。

## 11. PostgreSQL Docker 方案

开发环境使用 Docker Compose 启动 PostgreSQL。

`docker-compose.yml` 示例：

```yaml
services:
  postgres:
    image: postgres:16
    container_name: resumeai-postgres
    restart: unless-stopped
    environment:
      POSTGRES_USER: resumeai
      POSTGRES_PASSWORD: resumeai_dev_password
      POSTGRES_DB: resumeai_dev
    ports:
      - "5432:5432"
    volumes:
      - resumeai_postgres_data:/var/lib/postgresql/data

volumes:
  resumeai_postgres_data:
```

`.env.example`：

```env
DATABASE_URL="postgresql+psycopg://resumeai:resumeai_dev_password@localhost:5432/resumeai_dev"
VITE_API_BASE_URL="http://localhost:8000"
```

启动命令：

```bash
docker compose up -d
```

## 15. 关键原则

1. 严格前后端分离：前端 React + Vite，后端 FastAPI。
2. 前端不直接调用大模型。
3. 后端 API 只做入口，业务逻辑放到 Services。
4. Prompt 模板文件化、版本化。
5. 每次模型调用都写入 `prompt_runs`。
6. 首版聚焦简历生成闭环，不做复杂平台能力。
7. 所有生成内容必须基于用户真实输入，不虚构经历和数据。
8. 先用普通服务编排，不急于引入复杂多智能体框架。
9. 数据结构要为后续面试、评估、公司信息模块预留扩展空间。
