# 简历资料模型设计

## 1. 目标

简历助手不应在用户只输入一句经历后立刻生成空泛初稿。系统需要先收集足够的简历资料，再生成可展示的专业初稿。

核心原则：

- 基础信息决定简历是否像简历。
- 经历信息决定简历是否有说服力。
- 目标岗位决定简历内容如何取舍和表达。

## 2. 顶层资料模型

```json
{
  "target": {},
  "profile": {},
  "education": [],
  "skills": {},
  "experiences": [],
  "readiness": {}
}
```

## 3. 目标岗位信息

```json
{
  "company": "腾讯",
  "position": "数据分析实习生",
  "jobDescription": "负责 SQL 取数、业务数据分析、报表搭建和分析报告输出。",
  "keywords": ["SQL", "Python", "指标分析"],
  "coreAbilities": []
}
```

## 4. 基础信息

```json
{
  "name": "姓名",
  "phone": "电话",
  "email": "邮箱",
  "city": "城市",
  "school": "学校",
  "major": "专业",
  "degree": "学历",
  "graduation": "毕业时间"
}
```

MVP 必填建议：

- 姓名
- 电话或邮箱
- 城市
- 学校
- 专业
- 学历

## 5. 经历信息

```json
{
  "type": "project",
  "title": "课程问卷数据分析项目",
  "organization": "课程项目",
  "period": null,
  "role": "数据分析负责人",
  "background": "围绕用户满意度和功能偏好进行问卷分析",
  "tasks": [],
  "actions": [],
  "tools": [],
  "methods": [],
  "metrics": [],
  "results": [],
  "impact": [],
  "keywords": []
}
```

每段经历最低可生成标准：

- 经历名称不为空
- 角色或个人职责不为空
- 关键行动至少 2 条
- 工具或方法至少 1 条
- 结果或产出至少 1 条
- 有数据、规模、反馈、报告、奖项中的至少 1 个证据

## 6. 完整度判断

```json
{
  "resumeReadiness": "not_ready | draft_ready | strong_ready",
  "evidenceLevel": "low | medium | high",
  "missingFields": ["role", "methods", "results"],
  "nextQuestion": {
    "field": "role",
    "content": "你在这个项目中具体负责哪一部分？"
  }
}
```

## 7. 产品流程

```text
岗位分析
-> 填写基础信息
-> 输入经历
-> 结构化经历
-> 评估资料完整度
-> 不够则一问一答补信息
-> 达到 draft_ready
-> 生成简历初稿
-> 继续优化
-> 确认最终版
```
