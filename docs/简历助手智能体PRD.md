版本：v0.1  
状态：draft  
整理日期：2026-06-06  
资料来源：2026-06-02 面试模拟系统需求分析会议纪要  

## 1. 产品背景

AI 求职训练产品当前目标是从技术演示走向真实学生可用的产品版本。会议中明确提出，简历助手不是单纯生成漂亮 PDF 或套模板的工具，而应帮助学生围绕“具体公司 + 具体岗位”生成更有针对性的简历内容。

很多学生在写简历时存在三个典型问题：

- 不知道自己有哪些经历可以写。
- 不知道如何把经历表达成岗位需要的能力。
- 用同一份简历投递不同公司和岗位，缺少针对性。

因此，简历助手智能体应定位为“岗位定制型简历内容教练”，帮助用户挖掘经历、匹配岗位、优化表达，并为后续面试训练提供基础材料。

## 2. 产品目标

### 2.1 核心目标

帮助求职学生生成一份面向具体目标公司和目标岗位的高质量简历内容。

### 2.2 阶段目标

首版不追求完整平台化能力，也不优先追求复杂模板系统，而是先完成一个真实学生可以使用的简历内容生成闭环：

```text
输入目标公司和岗位
-> 分析岗位核心能力
-> 挖掘用户经历素材
-> 生成岗位定制简历内容
-> 进行内容评分和优化
-> 输出可继续修改的简历版本
```

### 2.3 非目标

- 首版不以生成精美 PDF 为核心目标。
- 首版不承诺自动获取未公开公司信息。
- 首版不做复杂招聘网站接口和爬虫集成。
- 首版不追求覆盖所有行业和所有岗位的专家级知识库。

## 3. 目标用户

### 3.1 主要用户

正在求职、实习申请、校招申请或转专业求职的学生用户，尤其是缺少简历写作经验的学生。

### 3.2 用户特征

- 有一些课程、项目、实习、竞赛、科研或校园经历，但不知道如何写进简历。
- 对目标岗位的能力要求理解不深。
- 简历表达偏泛化，缺少量化结果和岗位关键词。
- 希望获得比模板更有价值的内容指导。

## 4. 使用场景

### 4.1 投递前生成简历

用户准备投递某家公司某个岗位，希望快速生成一份有针对性的简历内容。

### 4.2 原简历优化

用户已有一份简历，希望根据目标岗位进行改写、删减、排序和强化。

### 4.3 经历素材挖掘

用户觉得自己“没什么可写”，系统通过连续提问帮助用户发现可写经历，并提炼成简历表达。

### 4.4 面试前材料整理

用户希望从简历中提取项目亮点、能力证据和可用于面试讲述的经历材料。

## 5. 核心功能需求

## 5.1 目标公司与岗位输入

### 功能说明

用户在开始生成或优化简历前，需要输入目标公司和目标岗位。

### 输入内容

- 目标公司名称。
- 目标岗位名称。
- 岗位 JD，可选。
- 用户投递目的，例如实习、校招、社招、保研/项目申请等，可选。

### 产品要求

- 如果用户没有 JD，系统应基于岗位名称生成通用岗位能力模型。
- 如果用户提供 JD，系统应优先基于 JD 提取能力要求和关键词。
- 系统应明确告诉用户当前分析依据是“用户提供的 JD”还是“通用岗位模型”。

## 5.2 岗位能力分析

### 功能说明

系统根据目标岗位，提取该岗位需要的核心能力，并作为后续简历生成和评估依据。

### 输出内容

- 核心能力列表。
- 每项能力的重要程度。
- 对应简历中应体现的证据类型。
- 岗位关键词。

### 示例

产品运营实习岗位可能包含：

- 用户洞察与数据分析。
- 活动策划与执行。
- 内容表达与沟通协作。
- 产品理解与问题拆解。

## 5.3 经历素材挖掘

### 功能说明

系统通过对话引导用户补充可写经历，帮助用户从原始经历中提炼简历素材。

### 可挖掘经历类型

- 实习经历。
- 项目经历。
- 课程作业。
- 科研经历。
- 竞赛经历。
- 社团和学生工作。
- 志愿活动。
- 个人作品或自学项目。

### 引导问题方向

- 你做过什么事情？
- 你的角色是什么？
- 遇到了什么问题？
- 你采取了什么行动？
- 最后有什么结果？
- 有没有数据、规模、排名、反馈、效率提升等可量化信息？
- 这段经历能体现目标岗位的哪项能力？

### 产品要求

- 不要求用户一开始就提供完整简历。
- 对信息不足的经历，系统应继续追问关键细节。
- 系统应避免编造用户没有提供的事实。
- 对缺少数据的经历，可以提示用户补充可能存在的数据，但不能虚构数据。

## 5.4 简历内容生成

### 功能说明

系统基于岗位要求和用户经历，生成可直接放入简历的内容。

### 生成范围

- 个人简介或求职摘要。
- 教育背景优化。
- 实习经历描述。
- 项目经历描述。
- 校园经历描述。
- 技能清单。
- 获奖与证书表达。

### 生成要求

- 围绕目标公司和目标岗位组织内容。
- 优先突出与岗位核心能力匹配的经历。
- 使用动作动词开头。
- 强调职责、行动、结果和能力证据。
- 尽量量化成果。
- 避免空泛表述，例如“认真负责”“沟通能力强”等。
- 保留用户真实经历边界，不虚构公司、项目、数据和头衔。

## 5.5 简历内容优化

### 功能说明

用户可上传或粘贴已有简历，系统对其进行岗位定制优化。

### 优化维度

- 岗位匹配度。
- 内容完整度。
- 表达清晰度。
- 结果量化程度。
- 经历排序。
- 关键词覆盖。
- 语言专业度。
- 冗余内容删除。

### 输出要求

- 给出总体评价。
- 标注主要问题。
- 给出修改建议。
- 输出优化后的版本。
- 对关键改动说明原因。

## 5.6 中英文简历支持

### 功能说明

支持中文简历、英文简历，以及中英文内容转换。

### 产品要求

- 中文简历强调准确、凝练、岗位匹配。
- 英文简历强调动词、结果、简洁表达和国际化简历习惯。
- 翻译时不能机械直译，应保持简历语境下的自然表达。

## 5.7 简历评分与差距反馈

### 功能说明

系统基于目标岗位能力模型，对当前简历进行评分和反馈。

### 评分维度

- 岗位匹配度。
- 经历含金量。
- 能力覆盖度。
- 表达质量。
- 量化程度。
- 真实性和可信度。

### 输出内容

- 总分或等级。
- 各维度评分。
- 已体现的优势。
- 缺失或薄弱的能力。
- 下一步补充建议。

## 5.8 版本管理与过程沉淀

### 功能说明

保存用户输入、系统分析、提示词思路和阶段性产出，便于后续迭代、复用和重构。

### 需要保存

- 目标公司和岗位。
- 岗位能力分析结果。
- 用户经历素材。
- 生成版本。
- 修改记录。
- 评分结果。
- 关键提示词和分析过程。

## 6. 用户流程

### 6.1 新建岗位定制简历

```text
用户输入目标公司和岗位
-> 用户粘贴 JD 或跳过
-> 系统分析岗位核心能力
-> 系统询问用户已有经历
-> 用户补充经历素材
-> 系统生成简历初稿
-> 系统评分并指出问题
-> 用户选择继续优化或导出内容
```

### 6.2 优化已有简历

```text
用户上传或粘贴原简历
-> 输入目标公司和岗位
-> 系统分析岗位要求
-> 系统诊断原简历问题
-> 系统追问缺失信息
-> 输出岗位定制优化版
```

### 6.3 单段经历打磨

```text
用户输入一段经历
-> 选择目标岗位能力
-> 系统追问细节
-> 生成 2-3 个不同风格版本
-> 用户选择并继续微调
```

## 7. MVP 范围

首版建议优先实现以下能力：

1. 输入目标公司、目标岗位和 JD。
2. 自动生成岗位核心能力和关键词。
3. 通过对话挖掘用户经历。
4. 生成中文简历核心模块内容。
5. 对已有简历进行岗位匹配度诊断。
6. 输出优化建议和修改后版本。
7. 保存关键输入、输出和分析过程。

首版可以暂缓：

- 精美 PDF 模板生成。
- 多模板在线编辑器。
- 招聘网站接口。
- 公司信息自动爬取。
- 完整账号体系和多人协作。

## 8. 页面与交互要求

### 8.1 首页或入口页

用户应能直接开始一个简历任务，而不是先看到技术调试界面。

核心入口：

- 生成岗位定制简历。
- 优化已有简历。
- 打磨单段经历。

### 8.2 岗位分析页

展示内容：

- 目标公司。
- 目标岗位。
- 岗位核心能力。
- 关键词。
- 简历建议突出方向。

### 8.3 经历挖掘页

交互要求：

- 以对话形式追问用户经历。
- 支持用户随时补充、跳过或修改。
- 将原始素材整理为结构化经历卡片。

### 8.4 简历生成页

展示内容：

- 生成后的简历内容。
- 每段内容对应的能力标签。
- 可编辑文本。
- 优化建议。
- 评分结果。

## 9. 智能体能力要求

### 9.1 分析能力

- 能理解岗位 JD。
- 能提取岗位能力和关键词。
- 能判断用户经历与岗位要求的匹配关系。

### 9.2 对话能力

- 能主动追问缺失信息。
- 能根据用户回答继续深入。
- 能保持简洁、清晰、鼓励式表达。

### 9.3 生成能力

- 能生成简历 bullet point。
- 能生成个人简介。
- 能生成中英文版本。
- 能根据不同岗位调整表达重点。

### 9.4 约束能力

- 不编造经历。
- 不虚构数据。
- 不夸大用户职责。
- 对不确定信息进行标注或追问。

## 10. 数据与知识要求

### 10.1 输入数据

- 用户基本信息。
- 目标公司和岗位。
- 岗位 JD。
- 用户原始简历。
- 用户补充经历。

### 10.2 后续可补充知识

- 岗位能力模型库。
- 行业简历样例。
- 中英文简历表达库。
- 常见学生经历改写样例。
- 公司公开信息库。

## 11. 验收标准

首版至少应达到：

- 用户能输入公司、岗位和 JD。
- 系统能输出岗位核心能力和关键词。
- 系统能通过提问帮助用户补充经历素材。
- 系统能生成面向岗位的简历内容，而不是泛化模板。
- 系统能指出简历与岗位之间的主要差距。
- 系统能给出可执行的修改建议。
- 系统默认展示真实用户界面，不暴露调试信息。
- 系统能保存关键提示词、分析过程和阶段性产出。

## 12. 风险与注意事项

1. 不要把简历助手做成单纯模板工具，内容质量应优先于排版。
2. 不要让用户承担过高输入成本，应通过追问逐步补齐信息。
3. 不要虚构经历、数据和成果，否则会损害用户求职可信度。
4. 不要过早做复杂接口和平台化能力，先完成可用闭环。
5. 不要忽视岗位差异，同一段经历在不同岗位下应有不同表达重点。
6. 不要丢失提示词、生成过程和版本记录，后续迭代会依赖这些沉淀。

## 13. 后续迭代方向

- 接入公司公开信息分析，进一步提升简历定制程度。
- 与面试模拟模块联动，从简历内容生成面试追问问题。
- 根据用户能力评估结果，推荐更适合的岗位方向。
- 支持英文简历深度润色和海外申请格式。
- 支持 PDF 或 Word 导出。
- 建立岗位能力模型和优秀简历样例库。
