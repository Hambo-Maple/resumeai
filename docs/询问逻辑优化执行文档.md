# 询问逻辑优化执行文档

整理日期：2026-06-19

## 1. 优化目标

当前简历助手需要从“先生成初稿，再继续优化”的流程，调整为“先充分收集资料，再生成可用版本”的流程。

核心变化：

- 删除“初稿”概念，不再引导用户过早生成简历。
- 保留“生成新版本”按钮，用于在资料收集完成后生成或刷新当前简历版本。
- 聊天框中不评价用户经历质量，只通过问题引导用户补充信息。
- 询问逻辑必须先分析用户已输入内容，再决定问什么，不能直接按固定框架硬塞问题。
- 逐个问题推进，一次只问一个明确问题，降低用户回答压力。

## 2. 新流程总览

```text
1. 用户输入基本信息
2. 系统分析目标公司、目标岗位、个人信息缺口
3. 系统引导用户补齐经历大框架
4. 系统逐个分析具体事例，并逐题追问细节
5. 用户点击“生成新版本”
6. 系统生成 Markdown 简历内容
```

## 3. 页面与按钮调整

### 3.1 保留内容

- 目标公司
- 目标岗位
- 岗位 JD
- 个人基础信息
- 经历内容
- 聊天框
- 生成新版本按钮

### 3.2 删除或弱化内容

删除产品文案和流程中的：

- 生成初稿
- 初稿
- draft
- v1 draft
- 先生成初稿再优化

如后端仍需要版本状态，可作为内部状态保留，但前端不向用户强调“初稿”。

### 3.3 “生成新版本”按钮语义

“生成新版本”表示：

- 当资料首次达到生成标准时，生成第一份 Markdown 简历版本。
- 当用户后续补充信息后，重新生成更新后的 Markdown 简历版本。

按钮可在资料不足时禁用，或点击后由系统继续追问缺失内容。

## 4. 对话原则

### 4.1 聊天框只问问题

聊天框中不输出类似以下评价：

```text
你的经历质量不足。
这段经历还不够有说服力。
目前素材无法支撑该岗位。
```

应改为引导式问题：

```text
为了更贴近这个岗位对数据分析能力的要求，你是否还有做过数据处理、报表、课程项目或活动复盘相关经历？
```

### 4.2 用户可以跳过

每个问题都允许用户回答：

```text
没有
暂时没有
跳过
不清楚
```

系统收到后：

- 不反复逼问同一个问题。
- 将该字段标记为 skipped。
- 继续分析下一个最重要缺口。

### 4.3 一次只问一个问题

禁止一次抛出大量问题：

```text
请补充背景、目标、角色、行动、工具、数据、结果和岗位关联。
```

应拆成逐个问题：

```text
这个项目里，你本人主要负责哪一部分？
```

用户回答后，再根据回答分析下一问。

## 5. 阶段 1：用户输入基本信息

该阶段用户可以通过手填或资料库填入：

- 目标公司
- 目标岗位
- 岗位 JD
- 姓名
- 电话
- 邮箱
- 城市
- 学校
- 专业
- 学历
- 毕业时间
- 技能
- 证书
- 奖项
- 已保存经历

完成后进入内容分析阶段。

## 6. 阶段 2：分析目标与个人信息缺口

### 6.1 分析范围

系统先分析以下内容是否缺失：

- 目标公司
- 目标岗位
- 岗位 JD
- 姓名
- 电话
- 邮箱
- 城市
- 学校
- 专业
- 学历
- 毕业时间
- 技能关键词
- 证书
- 奖项

### 6.2 提问规则

按简历生成必要性排序提问：

1. 目标岗位
2. 目标公司
3. 岗位 JD
4. 联系方式
5. 教育信息
6. 技能
7. 证书和奖项

每次只问一个缺失项。

示例：

```text
你这次主要想投递哪个公司？如果暂时没有具体公司，也可以回复“没有”，我会先按目标岗位来整理。
```

```text
这个岗位的 JD 方便粘贴一下吗？如果暂时没有，也可以回复“跳过”，我会先按岗位名称判断需要的能力。
```

```text
你的邮箱或手机号希望放在简历上吗？可以发其中一个；如果暂时不想填，也可以回复“跳过”。
```

## 7. 阶段 3：补齐经历大框架

该阶段只收集“有哪些事例”，不评价单个事例质量，不深挖细节。

### 7.1 分析目标

系统根据目标岗位和 JD，判断简历最好覆盖哪些经历类型。

例如数据分析岗位可优先覆盖：

- 数据分析项目
- SQL 或 Python 相关项目
- 报表或看板搭建
- 运营活动复盘
- 课程作业或比赛
- 团队协作经历

### 7.2 引导式提问

问题要帮助用户回忆，而不是只问“你有什么经历”。

示例：

```text
为了匹配这个岗位，你可以先想一想：有没有做过和数据、表格、问卷、报表、用户分析、活动复盘相关的事情？课程项目、社团活动、比赛、实习都可以。
```

如果用户只给出一段经历：

```text
这段经历我先记下了。除了这个项目，你还有没有其他能体现分析、沟通、执行或工具使用的经历？比如课程作业、社团活动、比赛、实习、小组项目都可以。
```

如果经历方向单一：

```text
目前我已经记录了数据分析相关经历。为了让简历更完整，你有没有能体现沟通协作、活动推进、报告展示或跨团队配合的例子？
```

### 7.3 大框架记录字段

每个事例先只需要记录：

```json
{
  "title": "经历名称",
  "type": "project | internship | course | competition | campus | volunteer | other",
  "organization": "组织/公司/课程/团队",
  "period": "时间",
  "role": "用户角色，可为空",
  "rawText": "用户原始描述",
  "detailStatus": "pending"
}
```

如果用户不知道名称，系统可临时生成一个中性标题，如：

```text
课程问卷数据分析项目
校园讲座活动运营
后端课程管理系统
```

## 8. 阶段 4：逐个事例追问细节

该阶段开始对每个具体事例进行分析和追问。

### 8.1 处理顺序

系统按岗位相关度排序经历：

1. 与目标岗位核心能力最相关的经历。
2. 有工具、方法、数据、结果线索的经历。
3. 能补齐岗位能力缺口的经历。
4. 其他辅助经历。

一次只处理一个事例。

### 8.2 先分析，再提问

每轮追问前，系统需要分析当前事例已有信息：

- 是否知道用户角色。
- 是否知道具体行动。
- 是否知道工具或方法。
- 是否有数据、规模或频次。
- 是否有结果、产出或反馈。
- 是否能对应目标岗位要求。

然后选择当前最关键的一个缺口提问。

### 8.3 追问字段

可追问字段包括：

- 背景或目标
- 用户本人角色
- 具体行动
- 使用工具或方法
- 数据、规模、频次
- 结果、产出、反馈
- 岗位相关能力

注意：这些字段是分析依据，不是固定提问顺序。

### 8.4 追问示例

角色缺失：

```text
在“课程问卷数据分析项目”里，你本人主要负责哪一部分？
```

行动缺失：

```text
你当时具体做了哪些动作？比如数据收集、清洗、统计、可视化、报告撰写里，哪些是你完成的？
```

工具缺失：

```text
这个项目里你用了哪些工具或方法？例如 Excel、Python、pandas、SQL、问卷星、Tableau，或者其他工具。
```

数据缺失：

```text
这个项目大概涉及多少数据或样本？比如问卷份数、用户数量、表格数量、活动人数，能估计也可以。
```

结果缺失：

```text
这个项目最后产出了什么？比如报告、展示、看板、结论、建议，或者老师/团队的反馈。
```

岗位关联缺失：

```text
如果投递这个岗位，你最希望这段经历体现哪种能力？比如数据分析、问题拆解、沟通协作、执行推进或工具使用。
```

### 8.5 事例完成标准

单个事例达到以下条件时，可标记为 detail_ready：

- 有清晰标题。
- 有用户本人角色。
- 至少 2 条具体行动。
- 至少 1 个工具或方法。
- 至少 1 个结果或产出。
- 至少 1 个数据、规模、频次或反馈证据。

如果用户多次跳过，允许以 weak_ready 标记，不继续追问。

## 9. 生成 Markdown 简历

当至少有一组资料达到可生成标准后，用户点击：

```text
生成新版本
```

系统生成 Markdown 简历内容。

### 9.1 Markdown 输出结构

建议结构：

```markdown
# 姓名

电话 | 邮箱 | 城市

## 求职目标

目标公司：xxx
目标岗位：xxx

## 教育背景

学校 | 专业 | 学历 | 时间

## 专业技能

- xxx
- xxx

## 项目/实习/校园经历

### 经历名称

组织/团队 | 角色 | 时间

- 使用动词开头描述行动、工具、数据和结果。
- 突出与目标岗位相关的能力。

## 证书与奖项

- xxx
```

### 9.2 生成原则

- 不虚构公司、岗位、项目、数据、奖项。
- 用户说“没有”或“跳过”的内容不编造。
- 可以优化表达，但不改变事实。
- 数据只能来自用户输入。
- 对于证据不足的经历，使用更保守表达。

## 10. 状态机建议

建议前端或后端对话状态使用：

```text
setup
target_profile_check
experience_outline_collection
experience_detail_collection
ready_to_generate
version_generated
```

状态说明：

- `setup`：用户填写基础信息。
- `target_profile_check`：检查目标公司、岗位、JD、个人信息缺口。
- `experience_outline_collection`：收集有哪些经历。
- `experience_detail_collection`：逐个经历追问细节。
- `ready_to_generate`：资料达到生成标准。
- `version_generated`：已生成 Markdown 简历版本。

```json
  }
```


## 12. 前端改造点

- 删除“生成初稿”按钮和相关文案。
- 保留“生成新版本”按钮。
- 将聊天区定位为“下一步问题”承载区。
- 问题区域需要显示当前正在补充的对象，例如：

```text
当前补充：课程问卷数据分析项目
```

- 用户回答“没有/跳过”时，前端应允许继续推进。
- 经历列表需要展示每段经历状态：

```text
待补充
补充中
可生成
已跳过
```

## 13. 验收标准

- 用户不点击“生成初稿”也能完整推进流程。
- 聊天框不会评价用户经历质量，只提出补充问题。
- 系统能先询问目标公司、岗位、JD、个人信息缺失项。
- 系统能引导用户回忆多个事例，而不是只处理第一段经历。
- 系统能逐个事例追问细节。
- 每次只问一个问题。
- 用户回答“没有”或“跳过”后，流程继续推进。
- 点击“生成新版本”后，输出 Markdown 简历内容。
- Markdown 内容不虚构用户未提供的信息。
