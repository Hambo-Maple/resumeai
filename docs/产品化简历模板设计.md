# 产品化简历模板设计

## 1. 设计目标

当前简历生成不能只停留在 MVP 的固定结构。产品化版本需要支持不同用户、不同岗位、不同经历类型下的动态简历生成。

核心目标：

- 生成结果必须像真实可投递简历，而不是经历分析或素材卡片。
- 简历正文只展示可投递内容，不展示系统诊断、缺口、建议或过程语。
- 模板可以根据岗位和经历类型动态排序、动态取舍。
- 后续可以承接用户信息库、多经历选择、多岗位定制和导出功能。

## 2. 当前模板问题

当前 `resumeDocument` 主要结构是：

```json
{
  "basics": {},
  "target": {},
  "summary": "",
  "education": [],
  "experience": [],
  "projects": [],
  "skills": {},
  "certificates": [],
  "awards": [],
  "additional": []
}
```

主要问题：

- `experience` 和 `projects` 过于固定，无法自然承载校园经历、科研经历、竞赛经历、志愿经历等。
- `summary` 容易生成空泛的个人简介，对学生简历不一定必要。
- 不同岗位的模块顺序不同，固定字段不利于动态排序。
- 前端需要为每类经历写固定渲染逻辑，后续扩展成本高。
- 后续信息库中会有多种经历素材，固定字段不适合做智能筛选和组合。

## 3. 产品化模板原则

### 3.1 简历正文原则

简历正文只允许出现：

- 用户真实提供的信息
- 基于真实信息整理出的专业表达
- 可被招聘方直接阅读的内容

简历正文禁止出现：

- 用户补充后
- 待打磨
- 仍需补充
- 缺少信息
- 建议补充
- 这段经历可以用于
- 岗位分析
- 经历卡片
- 系统判断

这些内容应该放在对话区、优化建议区或内部状态中。

### 3.2 动态模块原则

产品化简历不应该强制所有用户使用同一组经历模块。

系统应根据经历类型动态生成模块：

- 实习经历
- 工作经历
- 项目经历
- 校园经历
- 科研经历
- 竞赛经历
- 志愿/社会实践
- 其他经历

### 3.3 岗位适配原则

不同岗位可以采用不同模块顺序。

数据分析实习生示例：

```text
基础信息
求职意向
教育背景
专业技能
项目经历
实习经历
竞赛/校园经历
证书与荣誉
```

产品/运营实习生示例：

```text
基础信息
求职意向
教育背景
实习经历
项目经历
校园经历
专业技能
证书与荣誉
```

技术开发实习生示例：

```text
基础信息
求职意向
教育背景
专业技能
项目经历
实习经历
竞赛经历
证书与荣誉
```

## 4. 新版 resumeDocument 结构

建议升级为：

```json
{
  "basics": {
    "name": "",
    "phone": "",
    "email": "",
    "location": "",
    "links": []
  },
  "target": {
    "position": "",
    "company": "",
    "industry": "",
    "city": ""
  },
  "education": [],
  "skills": {
    "technical": [],
    "tools": [],
    "domain": [],
    "language": []
  },
  "sections": [],
  "certificates": [],
  "awards": [],
  "additional": []
}
```

相比旧结构，核心变化是：

- 用 `sections[]` 替代固定的 `experience[]` 和 `projects[]`。
- 用 `sections[].type` 表示经历类型。
- 用 `sections[].title` 表示前端展示标题。
- 用 `sections[].items[]` 表示该模块下的经历条目。

## 4.1 简历板块框架

产品化初稿必须先有明确的简历板块框架。

一份面向校招/实习/初级岗位的中文简历，建议采用以下标准框架：

```text
基础信息
求职意向
教育背景
专业技能
经历板块
证书与荣誉
其他信息
```

其中：

- `基础信息` 是必需板块。
- `求职意向` 是必需板块。
- `教育背景` 对校招/实习用户通常是必需板块。
- `专业技能` 对技术、数据、产品、运营等岗位通常是必需板块。
- `经历板块` 是简历核心，必须至少有一个可展示经历。
- `证书与荣誉` 有内容才展示。
- `其他信息` 有内容才展示。

经历板块不是一个固定模块，而是动态模块集合：

```text
实习经历
工作经历
项目经历
校园经历
科研经历
竞赛经历
志愿/社会实践
其他经历
```

前端最终展示时，不应该展示空板块。

## 4.2 板块级分析 SectionAnalysis

初稿生成时，不能只生成正文，还要对每个简历板块生成分析。

建议每个板块都有一个 `SectionAnalysis`：

```json
{
  "sectionKey": "skills",
  "title": "专业技能",
  "status": "usable",
  "importance": "high",
  "completeness": 70,
  "jobRelevance": 80,
  "contentStrength": 65,
  "truthfulnessRisk": "low",
  "summary": "已具备 Python、pandas、数据清洗和可视化报告相关证据，但 SQL 和可视化工具证据不足。",
  "gaps": [
    {
      "type": "missing_tool",
      "priority": "medium",
      "description": "缺少 SQL 或可视化工具的真实使用证据。"
    }
  ],
  "nextQuestion": "这个项目中除了 Python/pandas，是否还使用过 SQL、Excel、Tableau、Power BI 或 matplotlib？"
}
```

字段说明：

- `sectionKey`：板块标识。
- `title`：板块展示标题。
- `status`：板块当前状态。
- `importance`：该板块对目标岗位的重要性。
- `completeness`：信息完整度。
- `jobRelevance`：与目标岗位的相关性。
- `contentStrength`：内容强度。
- `truthfulnessRisk`：虚构或夸大风险。
- `summary`：给系统和建议区使用的板块分析。
- `gaps`：该板块缺口。
- `nextQuestion`：如果下一步要补这个板块，应该问用户什么。

`SectionAnalysis` 不进入简历正文，只进入优化建议、追问决策和质量报告。

## 4.3 板块状态

建议 `status` 支持：

```text
missing
insufficient
usable
strong
hidden
```

含义：

- `missing`：该板块应该有，但用户没有提供任何可用信息。
- `insufficient`：有少量信息，但不足以形成专业简历内容。
- `usable`：可以生成初稿内容，但仍有优化空间。
- `strong`：内容较完整，可以作为投递版本的重要支撑。
- `hidden`：该板块没有必要展示，例如用户没有证书或奖项。

## 4.4 各板块内容与分析规则

### 基础信息

正文内容：

```text
姓名
电话 | 邮箱 | 城市 | 链接
```

需要的信息：

- 姓名
- 手机号
- 邮箱
- 城市
- 作品链接/GitHub/个人主页，可选

板块分析重点：

- 是否缺少必要联系方式。
- 邮箱、电话是否格式合理。
- 链接是否适合目标岗位。

缺失处理：

- 简历正文可使用明显占位值。
- 优化建议区提示用户补充。
- 不允许 AI 编造真实联系方式。

### 求职意向

正文内容：

```text
求职意向：数据分析实习生
```

需要的信息：

- 目标岗位
- 目标城市，可选
- 目标行业，可选
- 目标公司，可选

板块分析重点：

- 岗位是否明确。
- 岗位是否和经历、技能匹配。
- 是否需要针对特定公司定制。

缺失处理：

- 如果岗位缺失，不能生成正式初稿。
- 如果公司缺失，可以生成通用岗位简历。

### 教育背景

正文内容：

```text
学校 | 学历 | 专业 | 时间
GPA / 主修课程 / 荣誉，可选
```

需要的信息：

- 学校
- 学历
- 专业
- 入学/毕业时间
- GPA，可选
- 主修课程，可选
- 校内荣誉，可选

板块分析重点：

- 校招/实习场景下教育背景是否完整。
- 专业是否与目标岗位相关。
- 是否有可展示课程、GPA、荣誉。

缺失处理：

- 对校招/实习用户，教育背景缺失应作为整份简历高优先级缺口。
- 缺失时不编造学校。

### 专业技能

正文内容：

```text
数据分析：Python、SQL、pandas
工具平台：Excel、Tableau、Power BI
业务能力：指标分析、用户分析、可视化报告
语言能力：英语 CET-6
```

需要的信息：

- 编程/工具技能
- 平台/软件
- 岗位相关方法
- 语言能力

板块分析重点：

- 技能是否有经历证据支撑。
- 是否覆盖目标岗位关键词。
- 是否存在虚构或堆砌软素质。
- 是否缺少岗位强相关技能。

缺失处理：

- 只展示有证据的技能。
- 对没有证据但岗位重要的技能，放入优化建议，不写入正文。

### 经历板块

正文内容：

```text
项目经历
课程数据分析项目 | 数据分析负责人 | 2025.03 - 2025.06
- 使用……
- 分析……
- 输出……
```

需要的信息：

- 经历名称
- 经历类型
- 组织/公司/课程/团队
- 用户角色
- 时间
- 关键行动
- 工具/方法
- 数据规模
- 结果/产出
- 岗位相关能力证据

板块分析重点：

- 经历分类是否准确。
- 经历是否能支撑目标岗位能力。
- bullet 是否具备动作、方法、对象、结果。
- 是否缺少角色、工具、数据、结果。
- 是否存在夸大贡献或编造成果。

缺失处理：

- 如果一段经历至少有背景、动作、工具或产出，可以进入初稿。
- 缺少角色、结果、指标时，进入优化建议和追问。
- 不能因为缺结果就把经历整体隐藏。

### 证书与荣誉

正文内容：

```text
英语 CET-6
校级二等奖学金
全国大学生数学建模竞赛省级二等奖
```

需要的信息：

- 证书名称
- 奖项名称
- 等级
- 时间，可选

板块分析重点：

- 是否与目标岗位相关。
- 是否有真实信息来源。
- 是否值得展示。

缺失处理：

- 没有证书/荣誉时不展示该板块。
- 不显示“暂无”。

### 其他信息

正文内容：

```text
作品集链接 / GitHub / 个人网站 / 其他可展示信息
```

需要的信息：

- 作品链接
- 开源项目
- 个人主页
- 其他无法归类但适合展示的信息

板块分析重点：

- 是否增强岗位匹配。
- 是否适合放在简历正文。
- 是否应改放到基础信息链接。

缺失处理：

- 没有内容时不展示。

## 4.5 初稿的板块级输出

产品化初稿建议不仅输出 `resumeDocument`，还输出 `sectionAnalyses`：

```json
{
  "resumeDocument": {},
  "sectionAnalyses": [
    {
      "sectionKey": "basics",
      "title": "基础信息",
      "status": "insufficient",
      "importance": "high",
      "completeness": 30,
      "jobRelevance": 60,
      "contentStrength": 30,
      "truthfulnessRisk": "low",
      "summary": "缺少姓名、电话、邮箱和城市，当前只能使用占位值。",
      "gaps": [
        {
          "type": "missing_contact",
          "priority": "high",
          "description": "缺少必要联系方式。"
        }
      ],
      "nextQuestion": "请补充你的姓名、电话、邮箱和所在城市。"
    }
  ],
  "experienceReports": [],
  "resumeQualityReport": {},
  "growthMap": {},
  "nextQuestion": {}
}
```

其中：

- `sectionAnalyses` 负责每个简历板块的判断。
- `experienceReports` 负责每段经历的判断。
- `resumeQualityReport` 负责整份简历的整体判断。
- `growthMap` 负责后续扩展路线。
- `nextQuestion` 负责当前最优追问。

## 5. sections 数据结构

```json
{
  "type": "project",
  "title": "项目经历",
  "items": [
    {
      "id": "experience_id",
      "name": "课程数据分析项目",
      "organization": "课程项目",
      "role": "数据分析负责人",
      "period": "2025.03 - 2025.06",
      "location": "",
      "bullets": [
        "使用 Python/pandas 对 300 份问卷样本进行数据清洗与整理，处理缺失值、异常值并形成可分析数据表。",
        "围绕用户满意度、使用频率和功能偏好等维度进行指标分析，提炼用户行为特征和功能使用差异。",
        "制作可视化报告并输出 3 个关键结论，支持小组明确后续产品优化方向。"
      ],
      "tags": ["Python", "pandas", "数据清洗", "指标分析", "可视化报告"],
      "confidence": "medium"
    }
  ]
}
```

字段说明：

- `type`：经历类型，用于排序、筛选和样式。
- `title`：模块标题，直接展示给用户。
- `items`：该模块下的经历条目。
- `id`：来源经历 ID，便于后续追问、更新和版本追踪。
- `name`：经历名称，例如项目名、公司名、活动名、课题名。
- `organization`：组织、公司、学校、课程、社团或团队。
- `role`：用户角色。
- `period`：时间。
- `location`：地点，可选。
- `bullets`：简历 bullet。
- `tags`：技能、方法、岗位能力标签。
- `confidence`：内容可信度或完整度，供内部使用，不一定展示。

## 7. 技能模块设计

产品化技能不建议只用一行文本拼接。

建议结构：

```json

  "domain": ["数据清洗", "指标分析", "用户分析", "可视化报告"],
```

```text
业务能力：数据清洗、指标分析、用户分析、可视化报告
```

## 9. 生成规则

### 9.1 信息来源

模型只能使用：

- 用户基础信息
- 用户教育背景
- 用户技能
- 用户结构化经历
- 岗位分析结果
- 用户后续追问回答

不得编造：

- 学校
- 公司
- 岗位
- 项目成果
- 上线结果
- 提升比例
- 奖项
- 证书
- 技术栈

### 9.2 经历分类规则

如果经历中出现公司、部门、岗位、实习、工作等信息：

- 优先放入 `internship` 或 `work`

如果经历中出现课程、项目、系统、分析报告、开发等信息：

- 放入 `project`

如果经历中出现社团、学生会、活动、校园组织等信息：

- 放入 `campus`

如果经历中出现论文、实验室、课题、导师等信息：

- 放入 `research`

如果经历中出现比赛、竞赛、建模、挑战赛等信息：

- 放入 `competition`

如果经历中出现志愿、公益、社会实践等信息：

- 放入 `volunteer`

### 9.3 bullet 生成规则

每条 bullet 应尽量满足：

```text
动作 + 方法/工具 + 对象/规模 + 结果/产出
```

示例：

```text
使用 Python/pandas 对 300 份问卷样本进行数据清洗与整理，处理缺失值、异常值并形成可分析数据表。
```

如果缺少结果，不允许编造，可以写：

```text
形成可视化分析报告，支持小组进行后续产品优化方向讨论。
```

不应写：

```text
显著提升用户满意度。
```

除非用户明确提供了真实数据。

## 10. 优化建议区

优化建议不属于简历正文。

建议区可以展示：

- 当前资料完整度
- 缺少的关键信息
- 下一步追问
- 简历质量诊断
- 建议补充的证据

示例：

```text
优化建议
- 需要补充项目中的具体角色。
- 需要补充最终报告是否被采纳或用于展示。
- 需要补充 SQL 是否真实使用过。
```

## 11. 对话追问区

对话区负责一次只问一个问题。

追问优先级：

1. 用户角色
2. 关键行动
3. 工具/方法
4. 数据规模
5. 结果产出
6. 业务影响
7. 岗位相关能力证据

追问示例：

```text
你在这个项目中具体负责哪一部分？是独立完成，还是和小组成员协作完成？
```

不要一次问多个问题。

## 12. 初稿生成产物设计

产品化初稿不应该只生成一个 `resumeDocument`。

初稿生成应同时产出：

```json
{
  "resumeDocument": {},
  "sectionAnalyses": [],
  "experienceReports": [],
  "resumeQualityReport": {},
  "growthMap": {},
  "nextQuestion": {}
}
```

这几个产物的职责不同，不能混用。

### 12.1 resumeDocument

`resumeDocument` 是当前可以展示给用户看的简历正文。

它只负责回答：

```text
基于当前已确认事实，现在能生成一份什么样的简历？
```

要求：

- 只展示可投递内容。
- 不展示缺口、建议、分析过程。
- 不编造用户没有提供的信息。
- 内容可以不完整，但结构必须专业。
- 缺失模块不强行展示。

### 12.2 SectionAnalysis

`SectionAnalysis` 是对简历某个板块的质量判断。

它回答的是：

```text
这个板块是否应该出现？
这个板块信息是否完整？
这个板块对目标岗位是否重要？
这个板块缺什么？
如果要补这个板块，应该问什么？
```

例如：

```json
{
  "sectionKey": "education",
  "title": "教育背景",
  "status": "missing",
  "importance": "high",
  "completeness": 0,
  "jobRelevance": 70,
  "contentStrength": 0,
  "truthfulnessRisk": "low",
  "summary": "校招/实习简历通常需要教育背景，但用户尚未提供学校、专业、学历和毕业时间。",
  "gaps": [
    {
      "type": "missing_education",
      "priority": "high",
      "description": "缺少学校、专业、学历和毕业时间。"
    }
  ],
  "nextQuestion": "请补充你的学校、专业、学历和毕业时间。"
}
```

使用场景：

- 决定某个板块是否展示。
- 决定某个板块是否进入优化建议。
- 决定下一步是补基础信息、教育背景、技能，还是经历内容。

### 12.3 ExperienceQualityReport

`ExperienceQualityReport` 是对单段经历的质量判断。

它回答的是：

```text
这段经历能不能写进简历？
这段经历适合放在哪个 section？
这段经历缺少什么事实？
这段经历还能继续问什么？
这段经历能支撑哪些岗位能力？
```

示例：

```json
{
  "experienceId": "experience_id",
  "experienceTitle": "课程数据分析项目",
  "sectionType": "project",
  "score": 62,
  "readiness": "draft_ready",
  "supportedAbilities": ["数据清洗", "指标分析", "可视化报告"],
  "gaps": [
    {
      "type": "role",
      "priority": "high",
      "description": "缺少用户在项目中的具体角色"
    },
    {
      "type": "result",
      "priority": "high",
      "description": "缺少分析报告中的关键结论"
    }
  ],
  "nextQuestion": "你在这个项目中具体负责哪一部分？是独立完成，还是和小组成员协作完成？"
}
```

使用场景：

- 判断某段经历是否能进入简历。
- 判断某段经历应该进入哪个 section。
- 为单段经历生成下一步追问。
- 更新该经历的 bullet 和证据强度。

### 12.4 ResumeQualityReport

`ResumeQualityReport` 是对整份简历的质量判断。

它回答的是：

```text
这份简历整体能不能投？
结构是否完整？
是否覆盖目标岗位核心能力？
模块顺序是否合理？
是否缺少关键模块？
是否存在虚构或夸大风险？
```

示例：

```json
{
  "overallScore": 58,
  "readiness": "draft_ready",
  "structureCompleteness": 60,
  "jobMatchScore": 55,
  "evidenceStrength": 50,
  "readabilityScore": 70,
  "truthfulnessRisk": "low",
  "missingSections": ["education", "internship"],
  "weakSections": ["project"],
  "coveredAbilities": ["数据清洗", "可视化报告"],
  "uncoveredAbilities": ["SQL取数", "业务复盘"],
  "globalGaps": [
    "缺少教育背景",
    "缺少目标岗位强相关 SQL 证据",
    "目前只有一段项目经历，整体经历厚度不足"
  ],
  "nextBestAction": "先补充教育背景和项目角色，再生成更完整版本"
}
```

使用场景：

- 判断整份简历是否达到初稿标准。
- 判断整份简历是否达到较强投递标准。
- 决定下一轮优先问基础信息、某段经历，还是新增经历。
- 判断是否可以引导用户确认最终版。

### 12.5 SectionAnalysis、ExperienceQualityReport 与 ResumeQualityReport 的区别

三者不能混在一起。

```text
SectionAnalysis
负责判断某个简历板块是否完整、是否展示、是否需要补。

ExperienceQualityReport
负责判断单段经历怎么补、怎么写、放在哪里。

ResumeQualityReport
负责判断整份简历是否完整、是否匹配岗位、是否达到投递标准。
```

关系：

```text
板块分析决定简历框架
↓
单段经历质量决定经历内容
↓
进入 resumeDocument.sections[]
↓
整份简历重新生成
↓
ResumeQualityReport 重新评估整体质量
```

### 12.6 GrowthMap

`GrowthMap` 是后续优化路线图。

它回答的是：

```text
这份简历后续还能新增什么？
应该新增到哪里？
补充后会增强哪一部分？
下一步最值得问什么？
```

示例：

```json
{
  "expandableItems": [
    {
      "sectionType": "project",
      "itemName": "课程数据分析项目",
      "canAdd": ["role", "methods", "businessInsights", "finalOutcome", "tools"],
      "reason": "当前项目已有工具和样本规模，但缺少个人角色和分析结论。",
      "nextQuestions": [
        "你在项目中具体负责哪一部分？",
        "报告中最后得出了哪些关键结论？",
        "报告是否用于课程展示或获得反馈？"
      ]
    }
  ],
  "potentialSections": [
    {
      "sectionType": "education",
      "title": "教育背景",
      "reason": "用户尚未提供学校、专业、学历和毕业时间。"
    },
    {
      "sectionType": "internship",
      "title": "实习经历",
      "reason": "目标岗位为实习岗位，如果用户有相关实习经历，应优先补充。"
    }
  ]
}
```

`GrowthMap` 不直接展示在简历正文中。

它可以用于：

- 生成优化建议。
- 生成下一步追问。
- 决定是否需要新增 section。
- 决定当前优化是改写已有 bullet，还是新增经历模块。

### 12.7 nextQuestion

`nextQuestion` 是系统当前最应该问用户的一个问题。

它应该来自：

- `SectionAnalysis` 中最高优先级的板块缺口。
- `ExperienceQualityReport` 中最高优先级的经历缺口。
- 或 `ResumeQualityReport` 中最高优先级的整体缺口。
- 或 `GrowthMap` 中最重要的新增方向。

要求：

- 一次只问一个问题。
- 问题必须能带来可写入简历的事实。
- 不要一次追问多个字段。

示例：

```json
{
  "target": "experience",
  "experienceId": "experience_id",
  "field": "role",
  "question": "你在这个项目中具体负责哪一部分？是独立完成，还是和小组成员协作完成？",
  "reason": "当前项目已有工具和样本规模，但缺少个人贡献，会影响 bullet 的可信度。"
}
```

## 13. 初稿生成标准

初稿不是最终稿。

产品化初稿的目标是：

```text
最小可读简历 + 可扩展简历骨架 + 缺口地图
```

初稿可以缺内容，但必须满足：

- 已有事实被专业表达。
- 简历结构清晰。
- 缺失信息不进入正文。
- 后续优化知道应该往哪里补。
- 可以区分“单段经历缺口”和“整份简历缺口”。

不合格初稿：

```text
只有一句泛泛描述。
把缺少信息写进简历正文。
把经历分析当成简历。
没有记录后续可扩展方向。
```

合格初稿：

```text
姓名
电话 | 邮箱 | 城市

求职意向
数据分析实习生

专业技能
数据分析：Python、pandas、数据清洗、问卷分析、可视化报告

项目经历
课程数据分析项目
- 使用 Python/pandas 对 300 份问卷样本进行数据清洗与整理，形成可用于后续分析的数据基础。
- 围绕问卷数据进行指标整理与分析，支持可视化报告制作。
- 制作可视化报告，沉淀项目分析结果并支持小组汇报或复盘。
```

同时生成：

```text
ExperienceQualityReport：这段项目缺少角色、结论、反馈。
ResumeQualityReport：整份简历缺少教育背景、实习经历、SQL 证据。
GrowthMap：后续可以补教育背景、项目角色、分析结论、工具细节。
nextQuestion：先问项目角色。
```

## 14. 旧模板迁移方案

旧结构：

```text
experience[]
projects[]
```

迁移到：

```text
sections[]
```

迁移规则：

- `experience[]` -> `sections[type=internship]` 或 `sections[type=work]`
- `projects[]` -> `sections[type=project]`
- `certificates[]` 保持不变
- `awards[]` 保持不变
- `additional[]` 保持不变

为了兼容旧数据，后端可以在一段时间内同时支持：

```json
{
  "experience": [],
  "projects": [],
  "sections": []
}
```

前端渲染优先级：

```text
如果存在 sections[]，优先渲染 sections[]。
否则回退渲染 experience[] 和 projects[]。
```

## 15. 实施顺序

建议按以下顺序实现：

1. 更新后端 schema，新增 `ResumeDynamicSection` 和 `ResumeDynamicItem`。
2. 更新 `generate_resume_document.v1.md`，要求模型输出 `sections[]`。
3. 新增 `ExperienceQualityReport`，用于判断单段经历质量。
4. 新增 `ResumeQualityReport`，用于判断整份简历质量。
5. 新增 `GrowthMap`，用于记录后续可扩展方向。
6. 更新后端 normalize 和 sanitize，兼容新旧结构。
7. 更新前端 `ResumeDocumentTemplate`，优先渲染 `sections[]`。
8. 更新诊断和追问逻辑，让问题能关联到整份简历或某段经历。
9. 用测试用例验证数据分析项目能生成专业简历。
10. 再扩展导出 PDF / Word。

## 16. 产品化验收标准

一版合格的产品化简历初稿应满足：

- 右侧简历正文没有过程语。
- 没有把建议、缺口、诊断写进简历。
- 没有编造用户未提供的结果。
- 至少包含基础信息、求职意向、教育背景、技能、一个经历模块。
- 经历模块名称专业，bullet 可直接阅读。
- 不同经历类型可以进入不同 section。
- 前端可以在不改代码的情况下展示新增 section 类型。
- 能生成单段经历质量报告。
- 能生成整份简历质量报告。
- 能区分“某段经历缺什么”和“整份简历缺什么”。
- 能给出后续可扩展方向，而不是只能润色已有 bullet。
