## 1. 核心判断

简历助手不应该只是“一键生成简历”的工具，而应该是一个**对话式简历教练**。

普通生成器的流程是：

```text
输入经历
-> 生成简历
```

但更适合本项目的流程应该是：

```text
输入经历
-> 系统追问
-> 用户补充
-> 系统更新经历素材
-> 再追问或生成简历
-> 用户继续对话修改
-> 形成最终简历
```

这样系统的价值不只是“改写文字”，而是帮助用户发现自己有哪些可写内容、哪些信息还缺失、哪些经历能证明目标岗位能力。

## 2. 产品定位调整

原定位：

```text
岗位定制型简历内容生成器
```

建议强化为：

```text
岗位定制型对话式简历教练
```

核心能力从“生成一份简历”升级为：

- 引导用户挖掘经历。
- 判断经历与岗位能力的匹配度。
- 通过追问补齐关键信息。
- 在信息充分后生成简历初稿。
- 通过对话持续优化简历。
- 最终形成用户确认过的简历版本。

## 3. 推荐用户流程

### 3.1 总流程

```text
岗位分析
-> 用户输入原始经历
-> 系统结构化经历
-> 系统判断信息是否足够
-> 如果不足，进入对话追问
-> 用户补充信息
-> 系统更新经历卡片
-> 信息足够后生成简历初稿
-> 用户继续通过对话修改
-> 系统生成新版本
-> 用户确认最终版
```

### 3.2 经历采集阶段

用户输入一段原始经历：

```text
我做过一个校园活动，负责宣传，最后来了 200 多人。
```

系统不应该立刻生成最终简历，而应先结构化已有信息：

```text
经历类型：校园活动
用户行动：负责宣传
已有结果：到场 200+ 人
缺失信息：宣传渠道、报名人数、到场率、用户个人角色、是否复盘
```

然后系统追问：

```text
1. 你用了哪些宣传渠道？
2. 报名人数和到场人数分别是多少？
3. 你有没有根据数据调整宣传策略？
4. 你在团队中具体负责哪一部分？
```

### 3.3 简历生成阶段

当经历信息相对充分后，系统生成第一版简历：

```text
ResumeDraft v1
```

这版是初稿，不是最终稿。它的作用是让用户看到内容雏形，并作为下一轮优化的对象。

### 3.4 对话优化阶段

用户可以用自然语言继续修改，例如：

```text
这段经历想更突出数据分析。
```

系统不应直接强行改写，而应判断是否缺少数据分析证据。如果缺少，应继续追问：

```text
你分析了哪些指标？
你用了什么工具？
你的分析结论影响了什么决策？
```

用户补充后，系统再生成新版本：

```text
ResumeDraft v2
```

### 3.5 最终确认阶段

当用户满意后，可以点击：

```text
确认最终版
```

系统将当前版本标记为：

```text
status = final
```

## 4. 初稿与最终稿的关系

### 4.1 初稿定位

初稿不是终点，而是评估和追问的基础。

```text
结构化经历
-> 简历初稿
-> 缺口诊断
-> 继续追问
-> 改写生成新版本
```

初稿可能存在：

- 数据不完整。
- 经历细节不足。
- 个人角色不清楚。
- 岗位关键词覆盖不足。
- bullet 偏保守或偏泛化。
- 某些能力没有足够证据支撑。

### 4.2 最终稿来源

最终稿不是一次生成出来的，而是通过多轮对话迭代得到：

```text
ResumeDraft v1 初稿
-> 用户补充信息
-> ResumeDraft v2 修改稿
-> 用户继续确认或调整
-> ResumeDraft v3 最终稿
```

最终稿应满足：

- 用户确认内容真实。
- 经历表达清晰。
- 岗位匹配度较高。
- 关键能力有证据支撑。
- 没有虚构数据和成果。
- 可直接复制到简历或进入导出流程。

## 5. 对话式优化的核心能力

### 5.1 缺口诊断

系统需要判断当前经历或简历缺什么。

常见缺口：

- 缺少用户个人角色。
- 缺少关键行动。
- 缺少方法和工具。
- 缺少量化结果。
- 缺少岗位能力证据。
- 缺少业务背景。
- 缺少成果影响。

### 5.2 主动追问

系统应根据缺口提出具体问题。

好的追问：

```text
你用了哪些宣传渠道？每个渠道大概带来了多少报名？
```

不好的追问：

```text
请补充更多信息。
```

追问应优先围绕：

- 用户角色。
- 关键行动。
- 方法工具。
- 数据规模。
- 结果反馈。
- 岗位能力证据。

### 5.3 经历卡片更新

用户回答追问后，系统应更新结构化经历，而不是只把回答追加到聊天记录。

更新对象包括：

- role
- background
- actions
- results
- metrics
- matchedAbilities
- missingInfoQuestions
- rewriteDirection

### 5.4 简历版本生成

每次重要补充后，系统可以生成新版本：

```text
ResumeDraft v1
ResumeDraft v2
ResumeDraft v3
```

版本之间应保留：

- 版本号。
- 生成依据。
- 修改说明。
- 是否最终版。

## 6. 建议数据模型

### 6.1 ResumeDraft 增强

当前已有 `resume_drafts` 表，建议后续增加：

```text
status: draft | revised | final
parent_draft_id: UUID | null
change_summary: text | null
```

含义：

- `draft`：初稿。
- `revised`：修改稿。
- `final`：用户确认的最终稿。
- `parent_draft_id`：当前版本基于哪一版改写。
- `change_summary`：本次修改说明。

### 6.2 ResumeConversation

建议新增：

```text
resume_conversations
- id
- task_id
- resume_draft_id
- status
- created_at
- updated_at
```

用于表示一次围绕简历生成或修改的对话会话。

### 6.3 ResumeMessage

建议新增：

```text
resume_messages
- id
- conversation_id
- role: user | assistant
- content
- intent
- created_at
```

`intent` 可选值：

```text
supplement_info
rewrite_request
clarification
system_question
final_confirm
```

这样后续可以追踪用户是如何一步步补充信息并形成最终稿的。

## 7. MVP 实现建议

不建议一开始把完整对话系统做得很复杂。MVP 可以先做简化版。

### 7.1 当前已实现

```text
岗位分析
-> 经历结构化
-> 简历初稿生成
```

### 7.2 下一步建议

把传统“评分按钮”调整为：

```text
继续优化
```

用户点击后，系统做两件事：

1. 诊断当前简历缺口。
2. 给出下一轮追问问题。

用户回答后：

```text
用户补充信息
-> 更新经历卡片
-> 生成新版本简历
```

### 7.3 MVP 简化流程

```text
生成简历初稿
-> 点击继续优化
-> 系统提出 3 个追问问题
-> 用户回答
-> 系统更新经历
-> 生成 ResumeDraft v2
```

暂时可以不做完整聊天系统，只要先支持一轮追问和改写。

## 8. Prompt 设计方向

后续可以新增或调整以下 Prompt：

### 8.1 diagnose_resume_gap.v1.md

用途：

```text
分析当前简历草稿相对于目标岗位还缺什么信息。
```

输出：

- 缺口类型。
- 缺口说明。
- 追问问题。
- 优先级。

### 8.2 update_experience_from_dialogue.v1.md

用途：

```text
根据用户对追问的回答，更新结构化经历卡片。
```

输出：

- 更新后的 actions。
- 更新后的 results。
- 更新后的 metrics。
- 更新后的 matchedAbilities。
- 仍然缺失的信息。

### 8.3 rewrite_resume_from_dialogue.v1.md

用途：

```text
基于用户补充信息和当前简历草稿，生成新版本简历。
```

输出：

- 新版本简历。
- 修改说明。
- 仍需补充的问题。

## 9. 产品体验原则

1. 不要让用户一次性填写复杂表单。
2. 系统应通过对话逐步获得信息。
3. 每次追问都要具体，避免泛泛地让用户“补充更多”。
4. 不要为了让简历好看而虚构事实。
5. 初稿应被明确标记为“初稿”，避免用户误以为已经完成。
6. 最终稿应由用户确认，而不是系统自动宣布完成。
7. 对话过程应更新结构化经历，而不是只生成聊天文本。
8. 简历版本应可追踪，方便用户回退或比较。

## 10. 结论

简历助手的核心竞争力不在于“一键生成”，而在于：

```text
通过对话追问，帮助用户挖掘真实经历中的有效证据，
再围绕目标岗位把这些证据组织成高质量简历。
```

因此，后续开发应从“生成器思路”转向“教练式对话思路”：

```text
岗位分析
-> 经历挖掘
-> 追问补充
-> 初稿生成
-> 对话优化
-> 版本迭代
-> 用户确认最终稿
```
