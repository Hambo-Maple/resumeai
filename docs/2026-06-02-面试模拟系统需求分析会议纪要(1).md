# 面试模拟系统需求分析会议纪要

整理日期：2026-06-02  
会议主题：AI 面试模拟系统需求分析与功能规划  
资料来源：会议录音转写文本  
状态：`draft`

## 一、会议背景

本次会议围绕 AI 求职训练产品中的“面试模拟 / 面试陪练”模块展开，重点讨论如何把当前原型从单纯的问答或追问能力，推进为可供真实学生使用、并可进一步展示给 HR 或合作方的产品化版本。

会议共识是：当前产品不应停留在技术演示层面，而应尽快形成一个面向真实用户的可用界面，让华师大等目标学生能够实际使用，并通过真实求职学生反馈继续迭代。

## 二、核心结论

1. 面试模拟系统要围绕“具体公司 + 具体岗位”展开，而不是泛泛地做面试问答。
2. 用户进入模拟面试前，应先看到目标岗位所需的核心能力，而不是只通过下拉列表选择。
3. 面试过程不能只有“追问”，还要支持多种提问方式组合，例如开放式提问、行为型提问、压力式提问、假设性提问和技能型提问。
4. 用户应能调整本轮训练重点，包括能力主题、提问方式比重和面试氛围。
5. 公司信息是面试准备的重要组成部分，系统应帮助用户理解目标公司、核心团队、公开信息和岗位相关背景。
6. 当前阶段先做可用版本，不要过早追求完整平台；各模块先做出来，再考虑衔接和重构。
7. 所有关键提示词、思路和阶段性产出都要保存，便于之后重做、迭代或整合。

## 三、产品需求整理

### 1. 面试前配置

用户在开始模拟面试前，应完成一组基础配置：

- 选择或输入目标公司。
- 选择或输入目标岗位。
- 查看该岗位所需核心能力。
- 选择本轮训练主题或能力重点。
- 调整不同能力维度的训练比重。
- 调整不同提问方式的比重。
- 选择面试氛围，例如正常、开放式、压力式等。

岗位能力展示不应只是下拉选择。更理想的方式是让用户一眼看到该岗位需要哪些核心能力，以及每项能力的重要程度。形式可以是图形化能力图谱，也可以是清晰的结构化列表，重点是可读、可比较、可选择。

### 2. 提问方式模型

会议明确指出，真实面试不可能只有追问。系统需要支持多种提问方式，并能根据训练目标进行组合。

建议首版覆盖：

- 行为型提问：例如 STAR 框架，要求用户描述情境、任务、行动和结果。
- 开放式提问：让用户自由表达观点、思路和判断。
- 压力式提问：测试用户在紧张、质疑或不确定条件下的反应。
- 假设性提问：给出场景，观察用户如何分析和决策。
- 技能型提问：直接考察专业能力、岗位知识或技术判断。

默认情况下，每类提问都应被适当覆盖；用户也可以根据自身短板或即将面对的面试类型，调高某一类提问的比例。

### 3. 公司信息准备

系统应帮助用户完成目标公司研究。会议中特别强调，面试官常会问“你对我们公司有了解吗”，如果用户无法回答，会明显减分。

公司信息模块建议覆盖：

- 公司基本信息，例如业务、规模、发展阶段、公开动态。
- 公司核心团队和关键人物背景。
- 与目标岗位相关的业务线或产品信息。
- 公司近期重要事件、融资、市场表现或公开机会。
- 若能查到，补充面试官或部门负责人的公开背景。

系统不需要承诺获取未公开信息，但应帮助用户充分掌握公开且重要的信息。重点不是堆资料，而是让用户能用自己的话复述公司情况，并能解释该公司与目标岗位的关系。

### 4. 模拟面试体验

模拟面试应尽量接近真实面试场景。

首版体验要求：

- 使用真实用户视角设计界面，不保留调试信息作为默认展示。
- 调试内容可以放入调试模式，仅供开发或测试使用。
- AI 面试官的声音应尽量贴近真实面试，后续可提供多种声音选择。
- 面试过程要支持连续追问和不同提问方式切换。
- 面试结束后，系统基于用户回答给出能力评分、差距判断和训练建议。

会议中对“声音”有明确关注：语气、语调和态度会影响真实面试压力感，因此后续语音选择与真实感是重要体验点。

### 5. 训练与反馈

系统不只是模拟一次面试，还要帮助用户知道下一步怎么练。

需要形成的闭环：

```text
选择公司和岗位
-> 查看岗位能力要求
-> 设置本轮训练重点
-> 进入模拟面试
-> 根据回答评估能力
-> 输出差距和训练方案
-> 支持下一轮专项训练
```

训练可拆为两类：

- 整轮模拟面试：模拟完整面试结构，适合面试前冲刺。
- 单项训练：针对某个能力、某类问题或某个短板反复练习。

## 四、相关模块扩展

会议后半段讨论了 AI 求职训练产品中与面试模块并行或衔接的其他模块。

### 1. 能力评估

能力评估应服务于岗位训练。它至少包含两层问题：

- 某个岗位需要哪些能力。
- 如何评估用户是否具备这些能力。

评估方式可以包括：

- 对话式提问。
- 题目或测验。
- 简历与项目经历分析。
- 语音表达和临场表现观察。

岗位能力模型初期可以先由大模型生成通用版本，后续再通过专业书籍、岗位资料和训练数据不断补强。

### 2. 简历助手

简历助手的重点不是生成漂亮 PDF，而是帮助用户生成面向具体公司和具体岗位的简历内容。

会议强调：

- 不应“一份简历打天下”。
- 应围绕目标公司和岗位组织简历内容。
- 需要帮助用户挖掘经历素材，因为很多学生不知道自己有哪些可写内容。
- 可支持中英文简历和格式优化，但内容质量优先于模板形式。

### 3. 职位与公司信息获取

职位推荐和公司信息获取可通过多种方式实现：

- 爬虫抓取公开岗位或公司信息。
- 对接招聘网站公开接口。
- 后续通过合作方或企业直接发布职位。
- 根据用户主观意愿和客观能力进行岗位排序和推荐。

会议建议当前阶段不急于把接口和爬虫全部做复杂，先保留方向，未来可与招聘网站、学校或企业合作对接。

## 五、阶段性行动项

| 行动项 | 负责人 | 要求 |
| --- | --- | --- |
| 面试模拟模块下一版 | 面试模块负责人 | 做到所有主要板块都能用，深度可以先弱一些 |
| 用户界面产品化 | 面试模块负责人 | 默认展示真实用户界面，调试信息进入调试模式 |
| 岗位能力展示 | 面试模块负责人 | 让用户一眼看到目标岗位核心能力和训练重点 |
| 提问方式配置 | 面试模块负责人 | 支持多类提问方式和比重调整 |
| 公司信息模块规划 | 相关负责人 | 整理公司基本信息、核心团队、岗位相关信息的框架 |
| 能力评估模块构思 | 相关负责人 | 梳理岗位能力模型和评估方式 |
| 简历助手模块构思 | 相关负责人 | 聚焦简历内容生成、经历挖掘和岗位定制 |
| 提示词与思路沉淀 | 全体 | 保存关键提示词、分析过程和阶段性产出 |
| 真实用户预览 | 面试模块负责人 | 下次给老师看前，先找有面试经验的学生试看 |

## 六、时间要求

会议中要求团队不要以“做好了再说”为节奏，而是先给出明确目标时间。

当前初步节奏：

- 下一个版本目标时间倾向于下周三前后。
- 如果本周四前能完成可用版本，可提前开会检查。
- PRD、分析文档或阶段性材料完成后，应先发出，不必等全部功能完成。

## 七、验收标准

下一版至少应达到：

- 主要模块可点击、可进入、可基本使用。
- 页面风格可以延续当前方向，但必须像真实用户会看到的界面。
- 不依赖开发者解释，用户能理解从选择岗位到开始训练的流程。
- 能展示岗位核心能力和训练重点。
- 能支持模拟面试的基本配置和开始流程。
- 能体现多种提问方式，而不是只有追问。
- 能给出训练后的基础反馈或后续训练方向。

## 八、风险与注意事项

1. 不要只做技术演示，要以真实用户可用为目标。
2. 不要过早把所有模块衔接得很复杂，先把各自可用版本做出来。
3. 不要把调试面板暴露给正式用户。
4. 不要只依赖泛化大模型输出，后续要补充面试、简历、岗位能力等专业知识材料。
5. 不要忽视语音真实感，声音、语气和压力氛围会直接影响训练效果。
6. 不要丢失提示词和思路记录，后续可能重构或从头再做，但这些沉淀会显著降低返工成本。

## 九、会议总结

本次会议将面试模拟系统从“能问问题的原型”推进为“可配置、可训练、可反馈的真实面试训练产品”。下一步重点不是追求完整平台，而是尽快做出一个真实学生能用、老师能看、后续可展示给 HR 的版本。

当前第一目标是：

> 让华师大的同学可以真正用起来。

围绕这一目标，面试模拟模块需要优先补齐岗位能力展示、训练重点配置、提问方式组合、公司信息准备和用户级界面体验。
