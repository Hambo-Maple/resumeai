# 简历模块化与整体分析分离设计

## 1. 背景

当前简历助手已经具备简历生成、对话追问、待确认修改、质量诊断等能力，但在产品化过程中暴露出几个关键问题：

- 已经写好的简历内容会在后续更新中被重新改写。
- 某段经历已经确认后，系统仍可能继续追问同一段经历。
- 用户回答过的问题可能在后续更新中丢失。
- 整体分析问题和具体模块追问混在一起。
- 用户不容易判断当前到底在修改哪一块内容。

因此，后续需要将简历从“一整篇文本”升级为“多个独立模块组成的结构化文档”。

## 2. 核心需求一：简历模块独立化

简历不应被视为一整篇可随意重写的文本，而应拆分为多个独立模块。

典型模块包括：

- 基础信息
- 教育背景
- 技能
- 项目经历 A
- 项目经历 B
- 实习经历 A
- 证书
- 奖项
- 其他信息

每个模块都应作为一个独立对象存在。

建议的数据模型：

```ts
type ResumeBlock = {
  id: string;
  type:
    | "basics"
    | "education"
    | "skills"
    | "project"
    | "internship"
    | "campus"
    | "research"
    | "competition"
    | "certificate"
    | "award"
    | "additional";
  title: string;
  content: unknown;
  qualityReport: BlockQualityReport;
  questions: BlockQuestion[];
  answerHistory: BlockAnswer[];
  status: "draft" | "editing" | "pending" | "completed" | "locked";
  locked: boolean;
  lastUpdatedAt: string;
};
```

## 3. 模块编辑规则

每次编辑必须绑定到一个明确模块。

核心规则：

1. 每次只编辑一个模块。
2. 每次只追问一个模块。
3. 每次只生成该模块的待确认稿。
4. 用户确认后只写入该模块。
5. 已完成或已锁定模块不会被自动修改。
6. 只有用户主动选择某个模块继续编辑，系统才允许解锁并修改该模块。
7. 其他模块即使参与整体分析，也不能被顺手改写。

模块生命周期建议：

```text
draft
  -> editing
  -> pending
  -> completed
  -> locked
```

当用户主动选择继续编辑时：

```text
locked/completed
  -> manual_edit
  -> pending
  -> completed/locked
```

## 4. 核心需求二：整体分析与模块修改分离

系统仍然需要做整份简历分析，但整体分析不能直接修改正文。

整体分析负责看全局，模块编辑负责动内容。

整体分析可以判断：

- 整份简历是否完整。
- 是否匹配目标岗位。
- 是否覆盖目标岗位需要的核心能力。
- 哪些模块太弱。
- 哪些能力缺少证据。
- 是否需要新增经历、证书、奖项或技能。
- 当前简历是否达到可投递状态。

整体分析的输出应是：

- 整体评价
- 能力缺口
- 弱模块列表
- 优先级建议
- 推荐下一步编辑哪个模块
- 是否可投递

整体分析不能直接改：

- 项目经历正文
- 技能列表
- 教育背景
- 已确认模块的表述
- 已锁定模块的排序或措辞

## 5. 推荐架构

```text
ResumeBlocks
  |
  |-- GlobalAnalyzer
  |     |
  |     |-- GlobalQualityReport
  |     |-- WeakBlockList
  |     |-- AbilityGapList
  |     |-- SuggestedActions
  |
  |-- BlockEditor
        |
        |-- selectedBlock
        |-- blockQuestions
        |-- blockAnswerHistory
        |-- pendingBlockDraft
        |-- blockQualityReport
```

整体分析器：

```text
只读所有模块
只输出诊断和建议
不修改正文
```

模块编辑器：

```text
只读取当前选中模块
只追问当前模块
只生成当前模块的待确认稿
只写入当前模块
```

## 6. 问题分类逻辑

问题必须先分类，再进入追问队列。

问题分为两类：

```text
module_question
global_question
```

模块问题用于完善某个具体模块或条目。

示例：

```text
这个项目是什么时间完成的？
你在这个项目中具体负责什么？
这个项目用了哪些工具？
这个项目最后有什么结果？
```

模块问题必须绑定：

```text
sectionKey + itemId + field
```

例如：

```text
project + local-experience-1 + tool
```

整体问题用于判断整份简历的能力覆盖、结构取舍和岗位匹配。

示例：

```text
除了这个项目，你还有其他 SQL 经历吗？
你是否还有更匹配数据分析岗位的经历？
你是否有证书、比赛或实习可以补充？
这份简历应该突出数据分析还是产品分析？
```

整体问题应绑定：

```text
global + field
```

例如：

```text
global + sql_evidence
global + ability_coverage
global + section_missing
```

## 7. 可写入标准

对于经历类模块，至少需要判断以下核心字段：

- 时间
- 角色
- 关键动作
- 工具或方法
- 结果或产出

如果这些核心信息仍缺失，则不应直接显示“更新到简历”。

如果用户选择跳过某个核心问题：

- 该问题标记为 skipped。
- 系统不再追问同一问题。
- 质量卡中显示“已跳过的问题”。
- 当前模块可以进入弱可写入版本，但不能被标记为强完成。

## 8. 用户确认逻辑

模块编辑时，用户回答追问后，系统生成待确认稿。

待确认稿不会立刻写入正式简历。

用户可选择：

- 更新到简历
- 继续优化这段
- 放弃本次修改

点击“更新到简历”后：

```text
写入当前模块
清空当前待确认稿
标记当前模块 completed
必要时锁定当前模块
清空当前问题
展示整体评价
等待用户主动选择下一步
```

系统不应自动继续修改该模块。

## 9. 预期收益

该设计可以解决以下问题：

- 已写好的模块不再被整体改写。
- 某段经历确认后不会被自动继续追问。
- 用户回答不会在连续追问中丢失。
- 整体分析与局部编辑边界清晰。
- 用户可以明确知道当前正在编辑哪个模块。
- 简历生成结果更稳定、更可控、更适合产品化。

## 10. 后续实现建议

建议分阶段实现：

1. 在前端状态层引入 `ResumeBlock` 模型。
2. 将当前 `resumeDocument` 转换为 `ResumeBlock[]`。
3. 将质量报告绑定到具体 `ResumeBlock`。
4. 将问题队列绑定到具体 `ResumeBlock`。
5. 将待确认稿从整份简历改为 `pendingBlockDraft`。
6. 确认写入时只更新对应 `ResumeBlock`。
7. 整体分析只输出 `GlobalQualityReport`，不直接修改模块内容。
8. 最后再将 `ResumeBlock[]` 渲染为完整简历预览。
